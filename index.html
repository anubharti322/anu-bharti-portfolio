SQL Workflow Walkthrough

Project Overview

Built a SQL-driven ETL that ingests messy Excel leads from multiple sources, standardizes and validates them, quarantines bad records, de-duplicates, and publishes a single trusted dataset for daily reporting.

Technical Implementation

  • Ingest → normalize → validate → reject/fix → publish.
  • Standardized phones, emails, dates, and budgets.
  • Required-fields check; invalid rows quarantined with reasons.
  • Duplicate control on normalized phone/email/ID.
  • Source lineage for accountability; automated daily runs.

Results and Impact

  • Approx. 88% reduction in manual cleaning/merging time; one time daily run process replaces ad-hoc work.
  • Reduction in duplicates by approx. 97%, improving accuracy of conversion and source metrics.
  • Data refresh moved to daily, speeding lead follow-ups and decision-making.
  • Better ownership and auditability through clear reject logs.
  • A consistent, analytics-ready dataset powering dashboards.

visualization Walkthrough

Overview

Built an end-to-end Power BI model on top of my cleaned SQL master table to deliver real-time insights on lead flow, agent productivity, conversion funnel, and SLA compliance.

Problem

  • Leads scattered across multiple Excel files, creating duplicates and stale reports.
  • Slow first responses and low follow-up completion (~65%) due to manual tracking.

What I Built

  • Power BI dashboards connected directly to SQL Server (staging → cleaned → master).
  • Automated refresh to keep visuals in sync without manual exports.

How It Works (Flow)

  1. Source: SQL cleaned/master tables from my ETL.
  2. Measures: DAX for conversion %, first-response time, duplicate rate, funnel stages.
  3. Visuals: Overview cards, funnel, agent/source leaderboards, area performance.
  4. Ops: Scheduled refresh for real-time insights.

Impact

  • Follow-up completion improved by 65% in two months.
  • ~32% uplift in lead→visit conversions through prioritization and timely nudges.
  • Duplicates reduced by ~95%, stabilizing KPIs and reducing noise.

Why It Matters

Turned reporting into a daily operating system—always-fresh, trustworthy metrics that let sales act faster on hot leads and focus on the highest-impact sources and agents.

Tech & Skills Demonstrated

  • DAX measures.
  • SQL Server connectivity, scheduled refresh.(optional RLS).
  • Performance tuning: measure tables, reduced high-cardinality visuals, no auto date/time.
  • Storytelling UX: bookmarks, drillthrough, tooltips, KPI cards, and funnel design.

Automation Process

Overview

I set up a daily SQL Server Agent job that runs my stored procedures end to end. It clears the staging table, imports new Excel files into raw_leads, normalizes and validates fields, routes incomplete records to leads_rejected with clear reasons, de-duplicates, upserts the final set into master_leads, and refreshes a lightweight reporting table for dashboards. The job is idempotent (safe to re-run) and keeps lineage (agent_name, source_file) for quick tracebacks.

Results & Impact

The pipeline now refreshes automatically each day, cutting manual prep to near zero and removing copy-paste errors. I have a single trusted reporting table, duplicates are handled consistently, incomplete rows are easy to fix, and dashboards load faster because the heavy cleaning happens in SQL before BI. Overall, it’s a stable, repeatable, auditable workflow that lets me focus on analysis instead of wrangling data.

Project Walkthrough

Overview

I turned a large road-accident dataset (~307k rows) into clear, business-ready insights. Starting from raw CSV, I cleaned and standardized fields, engineered time features (Month names, Hour), ran focused EDA, and built tidy visuals.

Insights and Interpretations

  • Seasonality: late-year months (esp. Nov/Oct) carry higher incident volumes → plan awareness drives & resource allocation accordingly.
  • Rush-hour risk: 3–5 PM is the riskiest window → target patrols, signage, and traffic calming measures around these hours.
  • Lighting matters: while Daylight dominates overall volume, no-lighting areas show substantial risk → prioritize lighting audits & maintenance in those zones.
  • Severity profile: majority Slight, but the presence of Fatal cases + label noise underscores the need for data governance (validation at source, controlled vocab).

    • Impact (Business Value)

  • Operational planning: tune staffing and checkpoints to peak months/hours.
  • Infrastructure: justify investments in lighting where accidents persist without illumination.
  • Governance: enforce category dictionaries (prevent “Fetal” typo), maintain duplicate/missing checks in the ingestion pipeline.
  • Communication: polished visuals for leadership decks; quick “so what?” callouts for non-technical stakeholders.
